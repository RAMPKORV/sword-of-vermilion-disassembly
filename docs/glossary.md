# Glossary — Sword of Vermilion Disassembly

Terms specific to this project, to 68000 assembly, and to Sega Genesis / Mega Drive hardware.
Cross-referenced with `constants.asm`, `notes.txt`, and the source modules.

---

## Hardware Architecture

**68000 (68k)**
Motorola 68000 CPU at the core of the Sega Genesis. 16/32-bit CISC processor.
Instructions are 1–5 words (2–10 bytes). All memory is big-endian.

**VDP (Video Display Processor)**
The Sega Genesis graphics chip (Yamaha YM7101 / custom). Manages VRAM, sprite table,
color RAM, H-scroll table, and all DMA transfers. Accessed via:
- `VDP_data_port` ($C00000) — data reads/writes
- `VDP_control_port` ($C00002) — command writes and status reads

**VRAM**
128 KB of Video RAM inside the VDP. Holds tile graphics (patterns), the two nametables
(Plane A, Plane B), the H-scroll table, and the sprite attribute table (SAT).
Not directly CPU-accessible — all VRAM access goes through the VDP ports or DMA.

**CRAM (Color RAM)**
64 entries × 9-bit color inside the VDP (4 palettes × 16 colors).
Addressed as 128 bytes. Written via VDP DMA or direct port writes.

**VSRAM (Vertical Scroll RAM)**
40 words of scroll data inside the VDP used for per-column vertical scrolling.

**DMA (Direct Memory Access)**
Hardware feature of the VDP that can copy data from ROM/RAM to VRAM/CRAM/VSRAM
at high speed, or fill a VRAM region with a constant byte. The 68000 halts during
DMA from ROM; RAM-sourced DMA can run during active display.

**SAT (Sprite Attribute Table)**
A table in VRAM (default at $7C00) with up to 80 sprite entries (8 bytes each).
Each entry specifies position, tile index, size, palette, and H/V flip.
In this game the SAT is built in RAM (`Sprite_attr_buffer`) and DMA'd to VRAM during VBlank.

**Z80**
Secondary 8-bit processor (Zilog Z80) used exclusively as a sound co-processor.
The 68000 controls it via:
- `Z80_bus_request` ($A11100) — request/release the Z80 bus
- `Z80_reset` ($A11200) — assert/release Z80 reset
- `Z80_ram` ($A00000) — 8 KB Z80 program/data RAM (shared bus)

**PSG (Programmable Sound Generator)**
Texas Instruments SN76489 chip. 3 square-wave tone channels + 1 noise channel.
Written via `PSG_port` ($C00011). The Z80 drives it in this game.

**FM (Frequency Modulation)**
Yamaha YM2612 FM synthesis chip. 6 FM channels + PCM. The Z80 drives it.

**PAL / NTSC**
PAL = 50 Hz (European), NTSC = 60 Hz (North American/Japanese).
Detected via bit 6 of `IO_version` ($A10001). The `IsPalRegion` macro wraps this check.

## Memory Map

**ROM**
$000000–$3FFFFF. The game binary. Labels with addresses in this range are code or data.

**RAM**
$FF0000–$FFFFFF (mirrors of $000000–$00FFFF in 68k address space, accessed as $FFXXXX.w).
All runtime data lives here. Constants in `constants.asm` define symbolic names.

**Stack**
`Stack_base` = $FFFFFE00. The stack pointer (A7) is reset here at the start of each
main-loop iteration.

**Object slots**
`Object_slot_base` = $FFFFD000. A fixed array of 40-byte object structs. The engine
iterates over these every frame. A5 is the conventional object pointer register.

**Palette buffer**
`Palette_buffer` = $FFFFC000. 128-byte RAM mirror of CRAM (4 palettes × 16 colors × 2 bytes).
DMA'd to CRAM once per VBlank.

**Sprite attribute buffer**
`Sprite_attr_buffer` = $FFFFE000. RAM mirror of the VDP SAT. Built during the frame,
flushed to VRAM during VBlank by `FlushSpriteAttributes`.

## VDP Concepts

**Nametable (Plane A / Plane B)**
A grid of tile attribute words in VRAM that specify which tile to display at each
background cell, plus palette index, priority, and H/V flip bits.
- Plane A: `$C000` (game graphics layer)
- Plane B: `$E000` (background / parallax layer)
- Window Plane: `$7000` (HUD / dialogue windows, not scrolled)

**Tile / Pattern**
8×8 pixels, 4 bits per pixel = 32 bytes per tile. Up to 2048 tiles fit in VRAM.

**Tile attribute word**
A 16-bit word in a nametable:
- Bits 14–13: palette index (0–3)
- Bit 15: priority (0 = behind sprites, 1 = in front)
- Bit 12: V-flip; Bit 11: H-flip
- Bits 10–0: tile index in VRAM

**DMA fill**
VDP can fill a VRAM region with a single byte value. Used in `DMAFillVRAM` macro.
Trigger: write fill byte to data port, then write DMA command to control port.

**VDP command longword**
A 32-bit value written to `VDP_control_port` to set the VDP read/write address and
memory type (VRAM/CRAM/VSRAM). Generated by the `vdpCmd` macro.
Example: `$40000003` = write to VRAM address $C000 (Plane A base).

## ROM Structure

**vermilion.asm**
Top-level file: ~30 `include` directives only. Defines the build order.

**header.asm**
ROM header (0x000000–0x000200): vector table, ROM ID string, memory map declaration.
Also includes `constants.asm` and `macros.asm`.

**init.asm**
`EntryPoint` → hardware init → `MainGameLoop`. The boot sequence and main dispatch loop.

**constants.asm**
All symbolic constants and RAM address definitions (~2976 lines).
Hardware registers, RAM addresses, game constants, object struct offsets, etc.

**macros.asm**
Assembly macros (~794 lines). See the section index at the top of the file.

**src/*.asm**
28 split source modules. Each opens with a `; ====...` section banner.
See `docs/module_boundaries.md` for per-module responsibilities.

## Assembly Conventions

**Label types** (see AGENTS.md for full table):
- `loc_XXXXXXXX` — undocumented location (hex ROM offset); should all be renamed
- `PascalCase` — function or data table
- `ALL_CAPS_WITH_UNDERSCORES` — symbolic constant
- `Mixed_Case` — RAM address or hardware register

**Size suffixes**
`.b` byte (8-bit), `.w` word (16-bit), `.l` long (32-bit).

**Addressing modes used frequently**
- `$NNNN.w` — short absolute (16-bit sign-extended address, used for $FFFCxxxx–$FFFFxxxx RAM)
- `$NNNNNNNN.l` — long absolute (used for ROM addresses and hardware registers)
- `NN(A5)` — object field access via object pointer
- `NN(A6)` — RAM base-relative access

**Register conventions**
- D0–D7: data / temporaries
- A0–A6: address / pointers
- A5: object/entity pointer (conventional, not enforced by hardware)
- A6: RAM base ($FFFFFF00 area, conventional)
- A7 (SP): stack pointer, reset to `Stack_base` each main-loop iteration

## Game Systems

**Program state machine**
Top-level control flow managed by `Program_state` ($FFFFC400).
Dispatched by `MainGameLoop` via `ProgramStateMap` in `states.asm`.
States: SEGA logo → title screen → name entry → prologue → in-game.
See `docs/module_boundaries.md` and `src/states.asm` header.

**Gameplay state machine**
In-game control flow managed within `GameplayState` (gameplay.asm).
States: town, overworld, dungeon (first-person), battle.

**Object system**
Fixed pool of 40-byte object slots at `Object_table_base` ($FFFFD000).
Each object has a type, flags, handler pointer, position, and sprite data.
Initialized from tables using `InitObjectsFromTable` / `initObjects` macro.
Field offsets are defined in `constants.asm` as `obj_*` constants.

**Script VM**
Bytecode interpreter in `src/script.asm`. Processes dialogue scripts stored in
`src/gamestrings.asm` and `src/strings.asm`. Opcodes are documented in
`tools/index/script_opcodes.json`.

**Event triggers**
Story progression flags at `Event_triggers_start` ($FFFFC720).
Each byte is 0 (not triggered) or $FF (triggered). The `script_cmd_triggers` macro
writes trigger flags from script bytecode.

**RLE compression** (map data)
Used for overworld sector maps. Encoding:
- Byte < $80: output as-is
- Byte ≥ $80: read next byte, output it ($80 − byte) times
See `src/mapdata.asm` header for full format.

**Kims**
The in-game currency. Stored as a BCD-encoded long at `Player_kims` ($FFFFC620).

**SRAM**
Battery-backed save RAM mapped at $200001 (odd bytes only). Managed by `src/sram.asm`.
Contains player stats, inventory, story flags, and map reveal state.

## Tooling

**verify.bat**
Builds the ROM and SHA256-checks it against the known-good hash.
Exit 0 = bit-perfect; exit 1 = mismatch. Always run after any change.

**tools/lint_*.js**
Node.js linters for common issues: raw RAM addresses, raw I/O addresses,
decimal literals in dc.*, duplicate constants, remaining loc_ labels, raw object offsets.
Run all at once with `node tools/run_checks.js`.

**tools/index/**
Generated index files: `symbol_map.json` (label → ROM address), `functions.json`,
`include_graph.json`, `script_opcodes.json`, `ram_symbols.json`.

**vermilion.lst**
Assembler listing file (generated by build, gitignored). Contains every instruction
with its ROM address, assembled bytes, and source location. Indispensable for
diagnosing bit-perfect failures — find the first diverging address to locate the bug.
See `docs/debugging_verify_failures.md`.
